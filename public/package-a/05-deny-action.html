Deny action — alongside Confirm and Reassign

A new staff action that closes the loop on trial requests the club can't accept: reason-coded, parent-notified, and MindBody-cancelled if a booking already opened.

What
Scope from the SOW: add a Deny action alongside the existing Confirm / Reassign. A reason picklist, a denial email with reason-specific copy, and auto-cancel in MindBody if a booking was made.
Why
Today unacceptable requests get silently ignored. Deny closes the loop politely, gives the parent useful context, and produces denial-reason data for pipeline reporting.
Who
Stuart builds the route + reason picker. Ibtissam owns the reason-specific email copy (spec 08) and the HubSpot workflow that branches on reason.

What's defined (per the SOW)

  • A third signed URL (alongside action: "confirm" and action: "reassign") with action: "deny", same HMAC scheme, same 24-hour expiry.
  • Clicking Deny opens a tiny HTML page — not an auto-apply — so staff can select a reason rather than clicking-and-regretting.
  • On submit, the Deal moves to stage Denied (from spec 02) and the picked reason lands on a Deal custom property (court16_denial_reason).
  • If a MindBody booking was already created (i.e. staff had previously confirmed and is now reversing), the handler cancels the MindBody enrollment before closing the Deal.
  • A reason-branching HubSpot workflow sends the parent an email tailored to why the request was denied.
  • Every Deny click is logged with correlation ID + chosen reason + staff token, matching the existing Confirm / Reassign audit trail.

Decide at kickoff

Ibtissam has domain context on what denial reasons actually come up in Court 16's operations. Cedarwind brings no firm proposals here — the picklist is her call.

The reason picklist

Concrete values + their internal names (court16_denial_reason dropdown options) get agreed at kickoff. Questions to resolve:

  • How many options? (Fewer is easier on staff; more is better for reporting.)
  • Is an "Other" + free-text staff-internal note needed for edge cases?
  • Which reasons need a reason-specific email vs. share a generic denial template? (Affects spec 08's email count.)

MindBody cancellation endpoint

MindBody's Public API v6 has /class/removeclientsfromclasses and /class/removefromwaitlist depending on whether a wait-list entry or a regular enrollment exists. Stuart confirms the correct call against the -99 sandbox once spec 04 has produced real enrollments to test against.

Token strategy

Two equivalent paths:

  1. Each Contact carries three signed URLs: court16_staff_confirm_url, court16_staff_reassign_url, court16_staff_deny_url.
  2. Each Contact carries one token; the staff-facing HTML pages branch on action at click time.

Option 1 matches the existing pattern (spec 01 already lists the two current URLs as Contact properties). Staying with that is the default; confirm at kickoff.
