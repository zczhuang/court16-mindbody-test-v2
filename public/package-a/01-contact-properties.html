Court 16 × Cedarwind · Package A · Spec 01

12 HubSpot Contact custom properties

The state-machine fields Cedarwind's app writes to every Contact alongside the existing form submission.

What
Twelve new custom properties on the HubSpot Contact object, keyed court16_*, covering booking state, MindBody IDs, signed staff-action URLs, and failure diagnostics.
Why
The booking app needs per-Contact state so workflows can react to status changes, staff can one-click confirm via signed URL, and failed bookings are traceable back to a correlation ID in under a minute.
Who
Ibtissam creates them in HubSpot (Settings → Properties → Contact properties). Estimated 15 minutes.

The twelve properties

All are single-line text unless otherwise noted.

Internal name Label Type Options
court16_correlation_id Court 16 correlation ID Single-line text Must have unique values: yes
court16_intent Court 16 intent Dropdown kid_trial, adult_intro
court16_booking_status Court 16 booking status Dropdown pending_staff, pending_payment, pending_staff_assist, confirmed, failed, duplicate_email_softwall, manual_review
court16_class_id Court 16 class ID Single-line text MindBody class ID as string
court16_location_slug Court 16 location slug Single-line text brooklyn, lic, fidi, ridgehill, fishtown, newton
court16_waiver_version Court 16 waiver version Single-line text e.g. v1.0
court16_offer_key Court 16 offer key Single-line text Adult-intro offer, e.g. tennis-intro-75
court16_mindbody_parent_id Court 16 MindBody parent ID Single-line text —
court16_mindbody_child_id Court 16 MindBody child ID Single-line text —
court16_staff_confirm_url Court 16 staff confirm URL Single-line text Signed, 24h expiry
court16_staff_reassign_url Court 16 staff reassign URL Single-line text Signed, 24h expiry
court16_admin_retry_url Court 16 admin retry URL Single-line text Signed, 24h expiry
court16_failure_reason Court 16 failure reason Multi-line text Populated when status = failed or manual_review
That's 12 new properties. One of the three URL fields (court16_staff_confirm_url) is the single trigger for Workflow 1 (staff notification). The other two are passed through to the same email body so staff has one-click access to every action.

Required Private App scopes

The HubSpot Private App that holds HUBSPOT_ACCESS_TOKEN needs:

  • crm.objects.contacts.read
  • crm.objects.contacts.write
  • crm.schemas.contacts.read — so the app can verify the 12 properties exist on boot

No custom-object scopes are needed. The Deal-pipeline work in spec 02 uses standard Deal scopes which are requested separately when that work lands.

Verifying the schema once created

After the properties are added, one curl call confirms all 12 exist:

curl -sS "https://api.hubapi.com/crm/v3/properties/contacts?archived=false" \
  -H "Authorization: Bearer $HUBSPOT_ACCESS_TOKEN" \
  | jq '.results | map(.name) | map(select(startswith("court16_")))'

Should print all 12 names. If any are missing, the app's first run will log a boot-time warning that names the gap.

Known limitation to note (Track 1)

One booking per Contact at a time. Each new submission overwrites the Contact's court16_correlation_id and court16_booking_status. Returning users are soft-walled before they ever reach the submit step, so this is effectively "new customer, one kid, one trial" scope for Track 1. The Deal pipeline in spec 02 is the path forward for multi-booking support.
