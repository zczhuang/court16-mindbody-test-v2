Parent MindBody password-setup email workflow

A HubSpot workflow that sends the parent a MindBody password-setup link shortly after their trial is requested, so their account is usable before the first class.

What
Scope from the SOW: a new HubSpot workflow that fires on Deal create (i.e. Deal enters the Trials pipeline in Requested Trial) and sends the parent an email containing a MindBody password-setup link.
Why
The booking app creates MindBody client records for parent + child as part of the submission, but the parent is never prompted to set a MindBody password. Without it they can't log in to see their booking, pay for enrollment, or reschedule.
Who
Ibtissam builds the workflow in HubSpot. Email template copy + MindBody URL format are decided with Stuart at kickoff.
Status
Greenfield · timing + URL format + copy decided at kickoff

What's defined (per the SOW)

  • Trigger: Deal enters the Trials pipeline in stage Requested Trial (i.e. the moment the booking app creates the Deal per spec 03). Matches the SOW phrasing "fired from HubSpot workflow on Deal create."
  • Classification: Transactional. The email is directly tied to the parent's booking action, not marketing.
  • Re-enrollment: Off. Each trial Deal qualifies once — a rescheduled or re-submitted booking doesn't re-fire the setup email.

Decide at kickoff

Send timing

Immediately on Deal create, or after a short delay? Tradeoffs:

  • Immediate → 1 email burst as the parent finishes submitting. Clear context, but risks feeling spammy if Workflow 2's acknowledgement also sends around the same time.
  • Short delay (30 min to a few hours) → Acknowledgement lands first, setup email lands separately. Feels more considered.

Ibtissam picks based on existing email-cadence norms at Court 16.

MindBody password-setup URL format

MindBody exposes a password-request flow per site. The exact URL shape + query parameters (site ID, email pre-fill) get locked in by Stuart at kickoff against the documented endpoint for Court 16's sites. Until then, the workflow template has a placeholder link that Ibtissam can swap in once Stuart confirms.

Email template

Copy draft lands in spec 08. Ibtissam tunes voice + CTA to match Court 16's existing parent-comms patterns.

Behavior when the email already has a MindBody password set

If a returning parent books a new trial, they already have a MindBody account with a password. The password-setup link in that case should no-op or show "you already have a password" rather than overwriting. Stuart confirms the exact MindBody behavior during implementation and flags if the email needs copy changes for that case.

Setup steps in HubSpot

  1. Marketing → Workflows → Create workflow → Based on a Deal.
  2. Enrollment trigger: pipeline is Trials, stage is Requested Trial.
  3. If a send delay is picked, add a Delay step.
  4. Send email → pick the password-setup template from spec 08.
  5. Recipient: Deal's associated Contact.
  6. Disable re-enrollment.
  7. Turn the workflow on.

Verification plan

  1. Submit a trial through the app (after specs 02 + 03 are live).
  2. Confirm the Deal lands in Requested Trial.
  3. HubSpot → Workflows → password-setup workflow → History: confirm the Contact enrolled.
  4. After any configured delay, confirm the email queued and sent.
  5. Click the link as the parent would → confirm MindBody loads the password-setup form for the right site + email.
  6. Submit a second trial from the same email (edge case — returning parent). Confirm re-enrollment doesn't fire a second setup email.
