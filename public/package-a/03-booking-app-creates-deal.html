Court 16 × Cedarwind · Package A · Spec 03

Booking app creates a Deal on submit

A code change in the booking app so every successful form submission produces a Deal in the Trials pipeline alongside the Contact it already creates.

What
Extend /api/book/trial/route.ts (kids trial flow) — and later, the equivalent intro path — so after the existing HubSpot Contact submission succeeds, the handler creates a Deal associated to that Contact at stage Requested Trial.
Why
Downstream workflows (specs 04, 05, 06, 07) react to Deal stage changes. Without a Deal per submission, the automation still has to hang off Contact-property changes, which works for one booking per Contact but breaks the moment a parent books a second trial.
Who
Stuart codes the change; sandbox-testable against the -99 site before kickoff.
Status
Greenfield · schema locked in with spec 02 at kickoff

How the flow runs today

The current kids-trial handler (app/api/book/trial/route.ts) does these steps in order:

  1. Validate the incoming payload (parent + child details, age band vs. class level).
  2. Call MindBody: look up the parent by email, create parent + child if new, wire the Guardian relationship.
  3. Submit to the existing Court 16 trial form via HubSpot's Forms v3 Integration endpoint. HubSpot creates-or-updates a Contact and copies the court16_* custom properties the app passes through.
  4. Return status: pending_staff to the client so the UI shows the "we got it" screen.

There is no Deal today. All booking state lives on the Contact.

A single new step: after the Forms API submission returns 200, the handler issues one POST /crm/v3/objects/deals that creates the Deal and associates it to the Contact by correlation ID.

What's defined

  • The Deal is created on the same booking-app server-side request, immediately after the Contact write succeeds.
  • Every Deal is associated to exactly one Contact (standard HubSpot Deal→Contact association).
  • Deals start in stage Requested Trial (per spec 02).
  • The correlation ID already threaded through every booking (today it's already on the Contact as court16_correlation_id) is the Deal's unique key too. Same value, same property name, on both objects — which is how spec 04's Confirm handler finds the right Deal.
  • HubSpot property uniqueness prevents duplicate Deals on retry — a re-submission with the same correlation ID returns the existing Deal.

Failure behavior

The MindBody write is already done by the time the Deal create runs, so a HubSpot 5xx at this step cannot roll back the MindBody booking. Three options to choose between:

  1. Log + continue + reconcile nightly with a sweep script.
  2. Retry inline with exponential backoff, fail the whole response if all retries fail.
  3. Queue the Deal create to a background job (new surface — more infra).

Cedarwind's recommendation is option 1: the Contact already carries the correlation ID + booking status, so the Deal is cosmetic until Confirm fires. Staff can still confirm via the Contact-workflow path if the Deal never lands. A short reconciliation script (under an hour to write) sweeps orphan Contacts daily.

Whether adult intros are in scope

The SOW lists "Modify booking app to create a Deal on form submit" — reads as kids-only unless otherwise decided. Adult intros (Tennis Intro, Pickleball Intro, Pickleball BOGO from Package B) currently land in HubSpot as Contacts only. Adding the Deal write there is cheap (~30 min of code) but not in the current SOW. Confirm at kickoff whether it's in or out.

Verification plan

  1. Sandbox-test against the -99 MindBody site with a fresh email.
  2. Submit a trial through the UI. Response should include the correlation ID.
  3. In HubSpot sandbox portal → Deals → Trials pipeline → find the Deal with that correlation ID in stage Requested Trial. The associated Contact should appear in the right-rail.
  4. Retry the same submission (same correlation). Confirm 200 with no second Deal created — property uniqueness rejects the duplicate.
  5. Temporarily break the HubSpot token to simulate a failure. Confirm the Contact still lands but the Deal create is logged as a failed trace step and the handler returns success to the client.
