Staff confirm moves Deal to Scheduled Trial

The existing one-click staff-confirm URL extends to move the associated Deal's stage, which is what workflows downstream hang off.

What
Extend /api/staff/confirm so after MindBody AddClientToClass succeeds, the handler also PATCHes the associated Deal from Requested Trial to Scheduled Trial, which is the signal HubSpot Workflow 2 (parent confirmation) + spec 07 (24h reminder) listen for.
Why
Today Workflow 2 fires on Contact property court16_booking_status changing to confirmed. Once the Deal pipeline is live, stage change is the cleaner and more extensible signal. It also makes the "Denied" and "No-Show" transitions natural extensions of the same pattern.
Who
Stuart codes the handler change. Ibtissam updates Workflow 2's trigger to listen on the Deal stage instead of the Contact property.
Status
Greenfield · trigger-change + PATCH wiring decided at kickoff

The confirm flow — end to end (as it'll run after the change)

  1. Parent submits the form → Deal lands in Requested Trial (per spec 03). Workflow 1 emails staff with three signed URLs: Confirm · Reassign · Deny.
  2. Staff clicks the Confirm URL. The URL is a signed JWT with a 24-hour expiry and action: "confirm" baked in — this part is already shipped in lib/staff-tokens.ts.
  3. /api/staff/confirm verifies the HMAC, rejects if expired, rejects if the action isn't confirm. Also already shipped.
  4. Handler looks up the Contact by court16_correlation_id. Idempotency: if court16_booking_status === "confirmed" already, the route returns "already confirmed" with no side effects. Also already shipped.
  5. MindBody AddClientToClass with ClientId from court16_mindbody_child_id and class from court16_class_id. Service ID threads through from TRIAL_CONFIG[locationSlug].trialServiceId when populated (Package B shipped that wiring).
  6. New in Package A. After MindBody succeeds, find the Deal with the matching court16_correlation_id and PATCH its stage from Requested Trial to Scheduled Trial. If no Deal is found (legacy Contact from before spec 03 shipped), skip the PATCH and log — don't fail the handler.
  7. Update the Contact's court16_booking_status to confirmed as belt-and-suspenders so any Contact-triggered legacy workflow still reacts. Already shipped.
  8. Return the "Confirmed" HTML response to the staff browser.

What's defined

  • The confirm URL scheme + HMAC signing + 24-hour expiry are already live in lib/staff-tokens.ts and don't change.
  • The single correlation ID is the join key: same value on the Contact and on the Deal, unique on both.
  • The MindBody write happens before the Deal stage move, so the physical booking is always the source of truth. If the Deal update fails, the parent is still booked.

Decide at kickoff

Workflow 2 trigger change

Today: "Contact property court16_booking_status is equal to confirmed." Proposed: "Deal stage changes to Scheduled Trial." Ibtissam confirms the switch at kickoff and reshapes the workflow.

Graceful-fallback strategy if the Deal PATCH fails

Options discussed:

  1. Log + continue; set court16_booking_status = confirmed on the Contact and return 200. Rely on the daily reconciliation script (spec 03) to catch the orphan.
  2. Fail the handler with 502 so staff sees an error + retries manually.

Cedarwind's recommendation: option 1. The booking is real in MindBody; one missed stage update shouldn't block the staff from moving on. Confirm at kickoff.

  4. Click the same URL again: expect "already confirmed" with no side effects (idempotency).
  5. Manually break the Deal association (delete the Deal in HubSpot) and trigger confirm on a fresh token: expect success, with a "no Deal found" note in the trace and MindBody + Contact both updated.
